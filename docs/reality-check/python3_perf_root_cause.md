# Python3 LD_PRELOAD Performance Regression Root Cause Analysis

**Bead:** bd-35hjg.2 (RC-WS1.2)  
**Date:** 2026-05-21  
**Status:** ANALYSIS COMPLETE

## Executive Summary

The python3 preload regression is caused by **excessive `gettid` syscalls** (61,726 calls consuming 92% of execution time) followed by a **SIGABRT crash** before completion. The issue affects both `strict` and `hardened` modes.

## Root Cause Identification

### Primary Issue: Excessive `gettid` Calls

**61,726 `gettid` syscalls** in a workload that should make near-zero such calls indicates:

1. **TLS access on every operation**: The library may be calling `gettid()` for thread-local storage indexing on every ABI entry point
2. **Missing TID caching**: Each TLS access triggers a fresh syscall instead of using a cached thread ID
3. **Hot-path instrumentation**: Safety checks or logging that call `gettid()` on every function entry

### Secondary Issue: SIGABRT Crash

The crash after excessive syscalls suggests:

1. **Assertion failure** in the membrane or ABI layer
2. **Memory corruption** from thread-safety issues
3. **Recursive re-entry** causing stack exhaustion

## Confirmed Code Locations

### Primary: Allocator Reentry Guard (`malloc_abi.rs:137`)

```rust
// malloc_abi.rs:135-138
#[inline]
fn current_allocator_reentry_slot() -> Option<&'static AllocatorReentrySlot> {
    allocator_reentry_slot_for_tid(raw_syscall::sys_gettid())  // <-- EVERY malloc/free/etc
}
```

This is called on **every allocation operation** (malloc, calloc, realloc, free, memalign) via `enter_native_reentry_guard()`. The code comment at line 71-76 explains why:

> "This path must not use Rust TLS or OwnedTlsCache: both can touch allocator/bootstrap machinery before malloc is safe."

### Secondary: OwnedTlsCache (`owned_tls_cache.rs:35`)

```rust
// owned_tls_cache.rs:34-35
pub(crate) fn with<R>(&'static self, f: impl FnOnce(&mut T) -> R) -> R {
    let tid = syscall::sys_gettid();  // <-- EVERY TLS access
    ...
}
```

By design avoids `__tls_get_addr` relocation but pays syscall cost on every access.

## Recommended Fixes

### Priority 1: Two-Phase TID Caching in Allocator

The allocator guard cannot use Rust TLS during bootstrap, but CAN cache after init:

```rust
// Add a static for cached TID per slot
static CACHED_TIDS: [AtomicI32; ALLOCATOR_REENTRY_SLOT_COUNT] = 
    [const { AtomicI32::new(0) }; ALLOCATOR_REENTRY_SLOT_COUNT];
static ALLOCATOR_INIT_COMPLETE: AtomicBool = AtomicBool::new(false);

#[inline]
fn current_allocator_reentry_slot() -> Option<&'static AllocatorReentrySlot> {
    // After init, use rseq or pthread_self() hash instead of syscall
    let tid = if ALLOCATOR_INIT_COMPLETE.load(Ordering::Relaxed) {
        // Fast path: use cached TID from __rseq_abi or similar
        get_cached_tid_fast()
    } else {
        // Slow bootstrap path: actual syscall
        raw_syscall::sys_gettid()
    };
    allocator_reentry_slot_for_tid(tid)
}
```

### Priority 2: Use `__rseq_abi` for TID (Zero Syscall)

Modern Linux (4.18+) provides `__rseq_abi` which contains the CPU ID and can derive thread identity without syscalls:

```rust
extern "C" {
    #[link_name = "__rseq_abi"]
    static RSEQ_ABI: RseqAbi;
}

fn get_tid_fast() -> i32 {
    // Fall back to syscall only if rseq unavailable
    unsafe { RSEQ_ABI.cpu_id_start as i32 }  // Or use the actual TID field if available
}
```

### Priority 3: Lazy Slot Acquisition

Instead of looking up slot on every call, acquire once per thread and store:

```rust
// Use a sentinel slot index in thread-local (arch-specific, no __tls_get_addr)
#[thread_local]
static mut MY_SLOT_INDEX: usize = usize::MAX;
```

### Priority 4: Investigate SIGABRT

After fixing the `gettid` storm, the crash may resolve. If not:
- Enable core dumps: `ulimit -c unlimited`
- Run under `gdb`: `gdb --args env LD_PRELOAD=... python3 -c 'print("done")'`
- Check for recursive mutex deadlocks or double-free

## Implementation Notes (bd-35hjg.3 — PARTIAL, NOT CLOSED)

### Fix Applied

`malloc_abi.rs` now derives a syscall-free thread key from the thread-control-
block self pointer (`fs:0` on x86_64, `tpidr_el0` on aarch64) and memoizes the
last `(tid, slot_index)` pair in a global `LAST_THREAD_CACHE`. The fast path
returns the cached slot only after `slot_matches_thread_key()` confirms the
cached slot is still owned by the current thread, so repeat lookups skip
`gettid` without the cross-thread slot-confusion bug of the first attempt.
Commits: `0e08e22a`, `52533362`, `97848ecf`.

### Verified Results (2026-05-21, HEAD with `97848ecf`, agent RedMill)

Re-verified directly: release `libfrankenlibc_abi.so` built via `rch`,
workload `python3 -c 'print("done")'`, min of 5 runs, `strace -f -c` for the
syscall breakdown.

| Metric              | Baseline | Strict        | Hardened        |
|---------------------|----------|---------------|-----------------|
| Wall time (min ms)  | 21       | 6559          | 29287           |
| Ratio vs baseline   | 1.0x     | **312x**      | **1394x**       |
| `latency_ratio_ppm` | —        | **312000000** | **1394000000**  |
| Exit code / signal  | 0        | 0 (no SIGABRT)| 0 (no SIGABRT)  |
| stdout              | `done`   | `done`        | `done`          |

**What the fix achieved:** the SIGABRT crash is gone — python3 now completes
correctly under both modes (behavior proof: stdout `done`, exit 0). The
`gettid` storm is cut from 61,726 to ~2,850 calls in strict mode.

**What the fix did NOT achieve:** the bd-35hjg.3 acceptance test — strict-mode
perf ratio within the `2000000` ppm smoke budget — **fails by 156x** (strict
312000000 ppm) and 697x (hardened 1394000000 ppm). The earlier claim that
timing "matches baseline (~210ms)" was wrong: it compared against the *pre-fix
crashed run* (SIGABRT at 109ms), which never executed the full python3 startup.
Once python3 actually completes, the real cost is visible.

### Where the residual cost is — NOT syscalls

`strace -f -c` of the strict run shows total syscall time of only ~4.8ms across
3,831 calls. The ~6.5s strict wall time is therefore **userspace membrane
overhead**, not syscalls. The original RCA pinned `gettid` (92% of a *crashed*
run's syscall time) as the dominant cost; that figure was an artifact of
profiling an incomplete run. The dominant cost of a *completed* python3 startup
under the membrane is per-libc-call validation overhead.

### Remaining work (bd-35hjg.3 stays OPEN)

bd-35hjg.3 as scoped ("algorithmic fix to the hot kernel named by the RCA") is
satisfied for the named `gettid` kernel, but the parent epic goal (650x → within
budget) is **not** met. Closing WS-1 requires a follow-up that profiles and
reduces the membrane per-libc-call userspace overhead — a different problem from
the `gettid` storm. The residual `gettid` callers (`util.rs:179`
`AbiReentrantMutex`, `io_internal_abi.rs:1515`, `owned_tls_cache.rs:35`) are now
a rounding error (<5ms total) and not worth optimizing until the userspace
overhead is addressed.
